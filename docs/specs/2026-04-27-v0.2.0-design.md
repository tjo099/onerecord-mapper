## Executive summary

v0.2.0 makes `@flaks/onerecord` a **credible reference TypeScript
implementation** of the IATA OneRecord cargo data model 3.2 + API
spec 2.2.0. The library is shifting audience from "internal Tracks
B/C/D consumers" to "anyone in the cargo industry asking whether
this lib is faithful to the spec."

That credibility comes from two complementary investments:

1. **Conformance-by-construction** — close three of the five known
   spec deviations (#1 booking-flow shortcut, #2 graph-walk
   validation kinds reserved-but-not-emitted, #4 hand-pinned FSU
   fixture). After v0.2.0, `docs/spec-deviations.md` retains only #3
   and #5, both spec-correct on the wire.
2. **Conformance-by-evidence** — ship infrastructure that lets
   third parties verify the conformance claim: a Docker-orchestrated
   contract test suite against IATA's reference NE:ONE Server,
   property-based round-trip tests for all 32 cargo data model
   classes, and re-enabled assertion catalogue tests that cover
   forbidden-state-transitions, operation-field validation, and
   prototype-pollution resistance.

The release is **non-breaking** at the type-system level: existing
v0.1.x consumers (Tracks B/C/D) can adopt v0.2.0 with zero call-site
changes. The booking-flow correction ships as a parallel
`acceptBookingOptionStrict()` API; the existing `acceptBookingOption()`
gets `@deprecated` and continues working through v0.2.x. Removal is
deferred to v0.3.0.

## 2. Architecture

Three new modules + one fixture refresh + one tooling extraction.

### 2.1 `src/booking-flow/transitions.ts` (modified)

Existing `acceptBookingOption(opt: BookingOption): ParseResult<Booking>`
keeps its current behavior and is annotated `@deprecated`. New
`acceptBookingOptionStrict(opt: BookingOption): ParseResult<BookingOptionRequest>`
follows spec §5.4 STATE_DIAGRAM faithfully, returning the
intermediate `BookingOptionRequest` state instead of jumping
directly to `Booking`.

The `BookingOptionRequest` schema and codec already exist in v0.1.x
(`src/classes/booking-option-request/`); the new function only needs
to construct an instance from a `BookingOption` after the state
transition validation, with no schema additions.

### 2.2 `src/dispatch/` (new)

A new module providing graph-walk validation as an **opt-in layer
above** existing per-class deserializers (Decision A1). Per-class
deserializer behavior is unchanged.

API surfaces:

- **Factory option**: `createMapper({ graphWalk: 'strict' })` returns
  a mapper whose `deserialize.<Class>` runs graph-walk first.
- **Namespace**: `onerecord.dispatch.deserialize.<Class>(raw)` for
  consumers who don't want a stateful factory.

The graph-walk pass runs **before** per-class Zod evaluation and
emits the four `ParseError.kind` values currently reserved but not
emitted by per-class deserializers:

- `duplicate_id_in_graph` — two distinct nodes share an `@id` IRI
- `missing_id` — a node embedded in a position requiring an `@id`
  has none (per spec §3.2)
- `wrong_type_for_endpoint` — an embedded node's `@type` does not
  match the field's expected class (e.g. a `Shipment` node embedded
  where a `Piece` is expected)
- `missing_type` — a non-leaf node lacks a required `@type`

### 2.3 `test/contract/` (new directory)

- **`docker-compose.neone.yml`** — orchestrates IATA's reference
  NE:ONE Server (image source TBD during Phase 5; the IATA-Cargo
  GitHub org publishes one)
- **`*.test.ts`** — vitest contract tests hitting the running
  server. Per Decision B1, smoke + key-class coverage:
  - Round-trip of a synthesized Waybill, Shipment, Piece against the
    reference impl
  - Booking flow: BookingRequest → BookingOption →
    BookingOptionRequest → Booking, end-to-end
  - Subscription/notification basic happy path
  - ~10 contract tests total
- Wired into `bun run test:contract` (script already in
  `package.json`, currently unused)
- Re-enabled in `release.yml`'s NE:ONE step (currently commented
  out per v0.1.2)
- Wired into `nightly.yml` for daily runs

### 2.4 `test/fixtures/iata/xfsu-status-codes.json` (regenerated)

Replace the v0.1.x hand-pinned fixture (where `pinnedSha` is the
placeholder string `manually-pinned-v0.1.0`) with output from a real
run of the convert script (now living in the sibling tooling repo
per §1.2 cleanup). The real `pinnedSha` will be a GitHub commit SHA
on `IATA-Cargo/ONE-Record` that pinpoints the source xlsx version.

The drift-detection test from CHANGELOG Phase 16 / T78 (already in
the test suite) gains real upstream comparison once the SHA is
real.

### 2.5 `onerecord-xlsx-tools/` (new sibling repo)

Hosts `convert-iata-xlsx.ts` and its `xlsx` dependency. The main lib
no longer carries the vulnerability. The tool publishes its own
release artifacts (CSV/JSON) that the main lib's fixtures can be
verified against.

This is a structural change to the project: one library + one
tooling repo, instead of one library with a sketchy script in
`scripts/`. Decision: **keep the main lib focused on runtime
contract; tooling lives separately.**

## 3. Design decisions

### 3.1 Decision A — Graph-walk dispatcher: opt-in (A1)

Default behavior of bare `WaybillCodec.deserialize(raw)` and
`onerecord.deserialize.Waybill(raw)` is **unchanged** in v0.2.0.
Graph-walk validation is reachable only through:

- `createMapper({ graphWalk: 'strict' }).deserialize.Waybill(raw)`
- `onerecord.dispatch.deserialize.Waybill(raw)`

**Rationale**: same shape as Zod's `safeParse` vs `parse` — strictness
is opt-in, the type-system contract is preserved (consumers who
exhaustively handle 22 `ParseError` kinds work either way). Avoids
runtime-behavior surprises for existing consumers. The cost is
one new namespace + one factory option.

In v0.3.0, the default may flip to `graphWalk: 'strict'` after one
release cycle of the new mode being available; that decision is
deferred.

### 3.2 Decision B — NE:ONE contract test depth: smoke + key-class (B1)

Phase 5 ships ~10 contract tests covering Ring 1 classes (Waybill,
Shipment, Piece) and the booking flow against IATA's reference
NE:ONE Server. Comprehensive contract testing (32 classes × all
endpoints) is deferred to v0.3.0.

**Rationale**: catches the highest-impact "library can talk to
reference impl" class of bugs at ~15% of the engineering cost of
comprehensive coverage. v0.2.0 fits in M scope.

### 3.3 Decision C — `xlsx` resolution: extract to sibling repo (C1)

Move `scripts/convert-iata-xlsx.ts` and the `xlsx` devDep to a new
`onerecord-xlsx-tools/` sibling repository. Strips `xlsx` from the
main lib's devDeps entirely; CI's `bun audit` step starts passing.

**Rationale**: cleanest separation of concerns. The convert tool is
not part of the runtime contract or the test infrastructure that
gates releases — it generates a fixture once per upstream xlsx
version, then the fixture is committed and verified by drift
detection. Tooling does not belong in the main lib. SheetJS team
explicitly recommends this pattern for security-conscious downstreams.

### 3.4 Decision D — Non-breaking via deprecation (project-wide)

`acceptBookingOption(opt) → Booking` retains current behavior with
`@deprecated` JSDoc pointing at `acceptBookingOptionStrict`. Removal
in v0.3.0.

**Rationale**: Tracks B/C/D adopt v0.2.0 with zero call-site changes.
Two-cycle deprecation policy gives consumers a release cycle to
migrate at their own pace. Standard pattern across React, Next.js,
TypeScript itself.

## 5. Success criteria

v0.2.0 ships when **all** of these are true.

### 5.1 Conformance — three deviations resolved

- `docs/spec-deviations.md` lists only #3 and #5 with cosmetic-
  deviation notes.
- `acceptBookingOption` has `@deprecated`;
  `acceptBookingOptionStrict` returns `ParseResult<BookingOptionRequest>`
  and has round-trip tests via the intermediate state.
- All four graph-walk `ParseError.kind` values
  (`duplicate_id_in_graph`, `missing_id`, `wrong_type_for_endpoint`,
  `missing_type`) emit in at least 2 distinct test fixtures each.
- FSU fixture `pinnedSha` is a real GitHub SHA on
  `IATA-Cargo/ONE-Record` and drift-detection passes against the
  upstream xlsx.

### 5.2 Evidence — conformance is verifiable

- `bun run test:contract` runs against a docker-compose'd NE:ONE
  Server and passes.
- `bun run test:property` runs property-based round-trip tests for
  all 32 classes in under 60s.
- `test/deserialize-errors/` catalogue has 0 `it.skip` placeholders.

### 5.3 Operational — releases work end-to-end

- CI green on every commit (`bun audit` passes — `xlsx` removed from
  main lib).
- `release.yml` runs end-to-end on the v0.2.0 tag: signed tag
  verifies, SBOM generated, SLSA provenance attached, GitHub Release
  page created automatically.
- `npm install @flaks/onerecord` resolves to v0.2.0 with prebuilt
  `dist/`.

### 5.4 Consumer-visible

- Tracks B/C/D `bun install` succeeds with `^0.2.0` pin and existing
  call sites work (deprecation warnings on `acceptBookingOption` are
  advisory).
- README "Why use this library?" updated to mention strict mode.
- CHANGELOG `[0.2.0]` entry comprehensive enough that an outsider
  can read it and understand the value proposition.

## 6. Open questions / risks

### 6.1 NE:ONE reference image source

The IATA-Cargo GitHub org publishes a NE:ONE Server reference
implementation. The exact docker image identifier (`ghcr.io/...` or
`docker.io/...`) needs to be located during Phase 5 setup. If the
image is not publicly distributed as a container, we may need to
build it from source, which expands Phase 5 scope.

**Mitigation**: scout the NE:ONE distribution model in week 6
before committing to docker-compose orchestration. Fallback: pin to
a specific source-tag and build in CI.

### 6.2 IATA spec amendment outcome

If the parallel IATA spec amendment for §5.2/§5.4 reconciliation is
**accepted** during v0.2.0's cycle, the canonical spec changes such
that the existing `acceptBookingOption` shortcut becomes
spec-correct. The v0.2.0 deprecation strategy still ships (consumers
benefit from the explicit `Strict` variant), but the deprecation
note in the library should be softened.

**Mitigation**: write the deprecation JSDoc as "deprecated for
v0.3.0; pending IATA §5.4 reconciliation" rather than absolute.

### 6.3 Property-test arbitrary generation effort

Some classes have rich nested structures (Waybill embeds Shipment
embeds Piece). fast-check arbitraries for these may be non-trivial
to write correctly without producing graphs that exceed the safety
limits.

**Mitigation**: shared helper enforces depth/node bounds on
generated graphs; per-class arbitraries scope to the class's own
fields and reference embedded classes by IRI rather than embedding
the full nested object.

### 6.4 Graph-walk performance

Adding a graph-walk pre-pass to `dispatch.deserialize.<Class>` has
runtime cost. For large graphs (close to safety limits) this could
materially slow opt-in strict mode.

**Mitigation**: benchmark the dispatcher path (existing
`vitest bench` infrastructure can be extended). Establish a
performance budget — e.g. strict mode is allowed to be ≤ 2x the
non-strict path at the safety-limit ceiling. If exceeded,
short-circuit by reusing the per-class deserializer's pre-validate
walk for graph-walk's node enumeration.

### 6.5 Sibling repo bootstrapping cost

Extracting `convert-iata-xlsx.ts` to a separate repo is right
architecturally, but bootstrapping a second repo (CI, README,
package.json, etc.) takes ~half a day of overhead in Phase 1.

**Mitigation**: minimal scaffolding — Apache-2.0, README, single
script, minimal CI. No release process needed; the tool generates
artifacts that the main lib commits.
