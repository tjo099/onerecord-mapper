## Executive summary

v0.2.0 makes `@flaks/onerecord` the **first publicly verifiable
TypeScript implementation** of the IATA OneRecord cargo data model
3.2 + API spec 2.2.0. The library is shifting audience from
"internal Tracks B/C/D consumers" to "anyone in the cargo industry
asking whether this lib is faithful to the spec."

**Note on framing**: domain review surfaced that several behavioral
fidelity items remain after v0.2.0 (full booking-flow state machine,
IRI dereferenceability, blank-node handling, IRI canonicalization,
context resolution order, AccessDelegation contract testing). The
"reference TypeScript implementation" claim is a v0.3 aspiration —
v0.2.0 ships **publicly verifiable** but not yet **canonically
referenceable**.

That verifiability comes from two complementary investments:

1. **Conformance-by-construction** — close three of the five known
   spec deviations (#1 booking-flow shortcut — partial, see §2.1
   scope note; #2 graph-walk validation kinds — JSON-LD structural
   only, see §2.2; #4 hand-pinned FSU fixture). Document five
   additional deviations (#6 domain-semantic cross-node validation,
   #7 IRI dereferenceability, #8 blank-node handling, #9 IRI
   canonicalization, #10 context resolution order) discovered during
   review with explicit v0.3 deferral.
2. **Conformance-by-evidence** — ship infrastructure that lets third
   parties verify the conformance claim: a Docker-orchestrated
   contract test suite against IATA's reference NE:ONE Server
   (smoke + key-class + §6.5 Operations + Notification flow);
   property-based round-trip tests for **Ring 1+2 classes (8 of
   32)**, with Rings 3-5 deferred to v0.3 per scope-realism finding;
   re-enabled assertion catalogue tests covering forbidden
   state-transitions, operation-field validation, and prototype-
   pollution resistance.

The release is **non-breaking** at the call-site level. Existing
v0.1.x consumers (Tracks B/C/D) adopt v0.2.0 with zero call-site
changes. The booking-flow correction ships as a parallel
`acceptBookingOptionViaRequest()` API; the existing
`acceptBookingOption()` gets `@deprecated` and continues working
through v0.2.x. Removal is deferred to v0.3.0. Note that the
`onerecord.dispatch` namespace addition is non-breaking at the
call-site level but **does** change the public-API surface — the
existing API-surface snapshot test will require an explicit update
(planned in Phase 3).

## 2. Architecture

Three new modules + one fixture refresh + one tooling extraction.

### 2.1 `src/booking-flow/transitions.ts` (modified)

**Renamed from rev-0**: `acceptBookingOptionStrict` →
`acceptBookingOptionViaRequest` per architect review (the "Strict"
suffix collides with `serializeWaybillStrict`-style serializer naming
used across 32 classes).

Existing `acceptBookingOption(opt: BookingOption): ParseResult<Booking>`
keeps its current behavior and is annotated `@deprecated` with
explicit migration guidance:

```typescript
/**
 * @deprecated v0.2 deprecation; removal in v0.3.
 *
 * Skips the BookingOptionRequest intermediate state per spec §5.4
 * STATE_DIAGRAM. Use acceptBookingOptionViaRequest(opt) and chain
 * to your persistence layer's BookingOptionRequest→Booking
 * transition. If the IATA §5.4 reconciliation accepts the §5.2
 * shortcut, this annotation will be relaxed.
 */
```

New `acceptBookingOptionViaRequest(opt: BookingOption): ParseResult<BookingOptionRequest>`
returns the spec-correct intermediate state.

**Scope acknowledgment** (per domain review): the §5.4 STATE_DIAGRAM
has more states than just the BookingOption→BookingOptionRequest
hop. v0.2.0 only models that single transition. The carrier-side
acceptance (`acceptBookingOptionRequest`,
`rejectBookingOptionRequest`, `confirmBooking`) is documented as a
v0.3 expansion — see §1.3 deferrals. v0.2.0's `docs/spec-deviations.md`
will mark deviation #1 as "partially closed" with a v0.3 plan
reference.

**Latent bug fix** (per domain review): `transitions.ts:49` currently
hardcodes `canTransition('BookingOption', 'REQUEST_PENDING', 'accept')` —
the `from` state is fabricated rather than read from
`opt.optionStatus`. Fix during Phase 2.

The `BookingOptionRequest` schema and codec already exist in v0.1.x
(`src/classes/booking-option-request/`); the new function only needs
to construct an instance from a `BookingOption` after state
transition validation, with no schema additions.

### 2.2 `src/dispatch/` (new)

A new module providing graph-walk validation as an **opt-in layer
above** existing per-class deserializers (Decision A1). Per-class
deserializer behavior is unchanged.

**API surface**:

- **Factory option**: `createMapper({ graphWalk: true })` (revised
  from `'strict'` per architect review — boolean, not stringly-
  typed; "strict" is overloaded by `serializeWaybillStrict`-style
  branded serializers).
- **Namespace**: `onerecord.dispatch.deserialize.<Class>(raw)` for
  consumers who don't want a stateful factory.

**Read-side only** (per architect review): `dispatch` is for
deserialization. Serialization does not need graph-walk because
app-layer values are already constructed under the `JsonLd<T>`
brand contract. There is no `dispatch.serialize`.

**Single-walk architecture** (promoted from rev-0 §6.4 risk-mitigation
to architecture decision per architect review):

The graph-walk pass shares its traversal with the per-class
`preValidate` step rather than walking twice. Phase 3 includes a
**Phase 3a prep step**: extract `walkGraph(input, opts, visitor)`
from `src/safety/pre-validate.ts:51-116` so both per-class
deserializers and dispatch share one traversal. Dispatch passes a
`{nodeVisitor}` that collects `{@id, @type, parentField}` during the
existing recursive walk; cross-node analysis (duplicate detection,
endpoint-shape check) runs on the pre-collected map. Zero double
traversal, no API change to `preValidate`'s contract.

**Emission scope** (deviation #2 closure):

The graph-walk pass emits the four `ParseError.kind` values
currently reserved but not emitted by per-class deserializers:

- `duplicate_id_in_graph` — two distinct nodes share an `@id` IRI
- `missing_id` — a node embedded in a position requiring an `@id`
  has none (per spec §3.2)
- `wrong_type_for_endpoint` — an embedded node's `@type` does not
  match the field's expected class (e.g. a `Shipment` node embedded
  where a `Piece` is expected). Requires a static field→expected-type
  map; building this map for 32 classes × N fields is the largest
  hidden cost in Phase 3 (per skeptic review). Mitigation: derive
  the map from existing Zod schemas where possible.
- `missing_type` — a non-leaf node lacks a required `@type`

Domain review flagged that the four kinds capture **JSON-LD
structural integrity** but not **domain-semantic cross-node**
violations (AWB consistency, total-pieces, graph-level cardinality,
reference resolvability). v0.2.0 explicitly scopes deviation #2 to
structural integrity; domain-semantic validation becomes deviation
#6 in `docs/spec-deviations.md`, deferred to v0.3.

### 2.4 `test/fixtures/iata/xfsu-status-codes.json` (regenerated)

Replace the v0.1.x hand-pinned fixture (where `pinnedSha` is the
placeholder string `manually-pinned-v0.1.0`) with output from a real
run of the convert script (now living in the sibling tooling repo
per §1.2 cleanup). The real `pinnedSha` is the GitHub commit SHA
on `IATA-Cargo/ONE-Record` that pinpoints the source xlsx version.

**Drift detection refinement** (per architect review): drift detection
compares **SHA256 of the upstream xlsx blob** at `pinnedSha` against
the SHA stored alongside the fixture. Field-level reconstruction
lives only in the sibling repo. Main lib never imports `xlsx`.

**Code count** (per domain review): do not pin to "26 codes." The
2024-era working draft had 26; the 2025-07 endorsed standard or
later working drafts may have 28-32. Convert script derives count
from the xlsx; fixture asserts the count + specific codes against a
snapshot to catch upstream additions.

### 2.5 `onerecord-xlsx-tools/` (new sibling repo)

Hosts `convert-iata-xlsx.ts` and its `xlsx` dependency. The main lib
no longer carries the vulnerability. The tool publishes its own
release artifacts (CSV/JSON) that the main lib's fixtures can be
verified against.

**Vulnerability management** (per skeptic review): the `xlsx`
vulnerabilities (Prototype Pollution + ReDoS) move to the sibling
repo, they don't disappear. Sibling repo gets its own `bun audit`
gate in CI. The sibling tool runs only at fixture-regeneration time
(roughly: when IATA publishes a new xlsx revision) — narrower
exposure surface than a runtime dep.

**Bootstrapping cost** (~half a day in Phase 0/1): minimal scaffolding —
Apache-2.0, README, single script, minimal CI (`bun test` + smoke
of `convert-iata-xlsx`), no release process. The tool generates
artifacts that the main lib commits.

**SBOM impact** (signal in §5.3): main lib's SBOM no longer includes
SheetJS once Phase 1 lands. This is a measurable success signal.

## 3. Design decisions

### 3.1 Decision A — Graph-walk dispatcher: opt-in, boolean

Default behavior of bare `WaybillCodec.deserialize(raw)` and
`onerecord.deserialize.Waybill(raw)` is **unchanged** in v0.2.0.
Graph-walk validation is reachable only through:

- `createMapper({ graphWalk: true }).deserialize.Waybill(raw)`
- `onerecord.dispatch.deserialize.Waybill(raw)`

**Rationale**: same shape as Zod's `safeParse` vs `parse` —
strictness is opt-in, the type-system contract is preserved
(consumers who exhaustively handle 22 `ParseError` kinds work either
way). Avoids runtime-behavior surprises for existing consumers. The
cost is one new namespace + one factory option.

**Naming revised** (per architect review): boolean `graphWalk: true`
instead of stringly-typed `'strict'`. "Strict" was overloaded by
`serializeWaybillStrict`-style branded serializers; the boolean
makes the option self-explanatory and avoids semantic conflation.

In v0.3.0, the default may flip to `graphWalk: true` after one
release cycle of the new mode being available; that decision is
deferred.

### 3.2 Decision B — NE:ONE contract test depth: smoke + key-class + Operations + Notifications

Phase 5 ships ~15 contract tests covering Ring 1 classes (Waybill,
Shipment, Piece), booking flow (forwarder side), §6.5 Operations
(JSON-Patch round-trip), and Notification subscribe-and-receive
flow against IATA's reference NE:ONE Server. Comprehensive contract
testing (32 classes × all endpoints; AccessDelegation; LogisticsEvent)
deferred to v0.3.0.

**Rev-0 → rev-1 expansion**: domain review flagged that "smoke +
key-class only" misses §6.5 Operations and Notification delivery,
which are the *most-used* OneRecord API contracts in production. A
"reference impl" claim that doesn't exercise them is brittle. They
move from v0.3 into M-scope.

### 3.3 Decision C — `xlsx` resolution: extract to sibling repo

Move `scripts/convert-iata-xlsx.ts` and the `xlsx` devDep to a new
`onerecord-xlsx-tools/` sibling repository. Strips `xlsx` from the
main lib's devDeps entirely; CI's `bun audit` step starts passing.

**Rationale**: cleanest separation of concerns. The convert tool is
not part of the runtime contract or the test infrastructure that
gates releases — it generates a fixture once per upstream xlsx
version, then the fixture is committed and verified by drift
detection.

**Vulnerability disclosure** (per skeptic review): the vulnerabilities
move with the dep. Sibling repo's own CI runs `bun audit` and
documents the policy in its README. Acceptable risk because (a) the
tool only runs at fixture-regeneration time, (b) it doesn't ingest
untrusted input — only IATA-published xlsx files fetched by SHA.

### 3.4 Decision D — Non-breaking via deprecation (project-wide)

`acceptBookingOption(opt) → Booking` retains current behavior with
`@deprecated` JSDoc pointing at `acceptBookingOptionViaRequest`.
Removal in v0.3.0.

**Rationale**: Tracks B/C/D adopt v0.2.0 with zero call-site changes.
Two-cycle deprecation policy gives consumers a release cycle to
migrate at their own pace. Standard pattern across React, Next.js,
TypeScript itself.

**Narrative correction** (per domain review): rev-0 framed §5.2 and
§5.4 as "contradictory." They are not — §5.2's POST body for
`/accept` is precisely the `BookingOptionRequest` resource named in
§5.4. The deviation is *elision* (skipping the request resource),
not *contradiction-resolution*. This rev corrects the framing in
the deprecation JSDoc + CHANGELOG entry.

### 3.5 Decision E — API-surface snapshot test handling

Adding the `onerecord.dispatch` namespace changes the public API
surface even though no call site of v0.1.x consumers breaks. The
existing snapshot test in `test/api-surface/` will fail without an
update. Phase 3 includes the snapshot update and a separate test
that verifies v0.1.x call patterns (e.g. `onerecord.deserialize.Waybill(raw)`)
still resolve to the same function.

## 6. Risks (R1-R11)

Rev-0 had 5 risks (R1-R5). Rev-1 adds R6-R11 per skeptic and
domain review.

### R1. NE:ONE reference image source — *now closed by Phase 0*

Was rev-0 §6.1. Promoted to a kickoff gate (§1.2) because review
flagged this as a BLOCKER, not a Phase 5 mitigation.

### R2. IATA spec amendment outcome (BookingOption shortcut)

If parallel IATA spec amendment for §5.2/§5.4 reconciliation is
**accepted** during v0.2.0's cycle, the canonical spec changes.
Mitigation: deprecation JSDoc wording is conditional ("pending
IATA §5.4 reconciliation"), not absolute.

### R3. Property-test arbitrary generation effort

Some classes (Ring 1+2) have nested structures (Waybill embeds
Shipment embeds Piece). Mitigation: shared helper enforces depth/
node bounds; per-class arbitraries scope to the class's own fields,
reference embedded classes by IRI in some cases and by embedded
object in others to test both. Skeptic-flagged risk that the
embedding strategy choice introduces inconsistency across the 8
classes.

### R4. Graph-walk performance

Adding a graph-walk pre-pass to `dispatch.deserialize.<Class>` has
runtime cost. Mitigation: §2.2 single-walk architecture (visitor
pattern through `walkGraph`); §5.2 perf budget ≤2x at safety-limit
ceiling, CI-enforced.

### R5. Sibling repo bootstrapping cost (~half a day)

Mitigation: minimal scaffolding in Phase 0/1.

### R6. NE:ONE Server licensing — closed by Phase 0 (was BLOCKER)

Per skeptic review. Phase 0 confirms image is publicly distributable
and license-permits CI use BEFORE Phase 5 commits.

### R7. `convert-iata-xlsx.ts` correctness — closed by Phase 0

Per skeptic review. v0.1.0 fixture was hand-pinned, not script-
generated; the script may produce different codes when run against
the real xlsx. Mitigation: Phase 0 runs the script and diffs
produced codes against the hand-pinned fixture; any delta resolved
as a domain question (working-draft drift) before Phase 1.

### R8. Solo maintainer bus factor

Per skeptic review. Single maintainer, signing key on one machine,
no co-maintainer. Mitigation: `MAINTENANCE.md` drafted in Phase 0
documents the GPG/release process; revocation cert backed up
offline. Long-term: invite co-maintainer in v0.3+ once the project
has external traction.

### R9. Subagent style variance on Phase 4

Per skeptic review. 4 subagents producing fast-check arbitraries
will diverge in style. Mitigation: shared helper extracted FIRST
(the Waybill arbitrary refactored into the helper), then subagents
adopt the helper for their classes. Style consistency enforced by
helper signature, not by review.

### R10. API-surface snapshot break on dispatch namespace

Per skeptic review. Adding `onerecord.dispatch` is non-breaking at
call-site but breaks the `test/api-surface/` snapshot. Mitigation:
explicit Phase 3 task to update the snapshot + add a separate test
asserting v0.1.x call patterns still resolve to the same functions.

### R11. Manual `npm publish` from maintainer's laptop

Per skeptic review. v0.2.0 publishes via manual `npm publish` (per
§1.3 deferral of automated publish). Risk of wrong env, untagged
dist, accidental `--access restricted`. Mitigation: pre-publish
dry-run + clean-dir install simulation per the v0.1.2 playbook
(documented in `MAINTENANCE.md`).
